# CHARLIE Private Executive Master Plan

## Governing Doctrine

CHARLIE is Charl's private digital executive. CHARLIE is not CORE, a coding
agent, or a command menu. He gathers authoritative evidence across the
operating system, reasons against Charl's goals and preferences, delegates
work, monitors outcomes, and involves Charl only for genuine discretion or
protected authority.

The permanent hierarchy is:

```text
Charl
  -> CHARLIE private executive
       -> domain leaders and operational tools
       -> CORE engineering workforce
            -> Codex Builder and specialist engineering stages
```

- Charl owns values, risk appetite, strategic direction, and exceptional decisions.
- CHARLIE owns executive context, investigation, synthesis, delegation, follow-through, and concise communication.
- CORE owns scoped engineering planning, building, testing, review, and release.
- Codex is CORE's primary technical builder. It does not replace CHARLIE.
- CHARLIE does not write code; he delegates engineering outcomes to CORE.
- OOM SAKKIE, SAM, BEACON, HERDMASTER, ANALYST, FRED, finance, telemetry, and document services provide specialist evidence or bounded execution.
- Supabase operational records remain authoritative. Model memory is never operational truth.

## Non-Negotiable Behavior

1. Ordinary read questions never require approval.
2. CHARLIE investigates evidence gaps instead of guessing or returning a generic permission message.
3. CHARLIE may use several read tools in one turn and continue until the answer is materially complete or a real blocker is proven.
4. Engineering work is delegated to CORE by `mission_id`; CHARLIE does not edit application code.
5. Domain work is delegated to the owning agent or bounded service.
6. Every delegation has an outcome, acceptance condition, owner, next check, and durable commitment.
7. Routine reversible actions use delegated policy. Protected actions require Charl.
8. Red-zone authority is never inferred from model confidence.
9. CHARLIE reports business meaning, not raw system dumps.
10. Dashboard, Telegram, and voice use the same runtime, memory, policies, and tools.

## Delivery Phases

### Phase 1 - Doctrine And Contract

- Persist this doctrine in operations and Vault identity documentation.
- Define executive answer, evidence, delegation, commitment, and escalation contracts.
- Add drift tests that fail if CHARLIE is described as CORE or a coding agent.

Done when tests prove the hierarchy and authority wording.

### Phase 2 - Universal Evidence Fabric

- Register typed read capabilities for CORE, pigs/litters/weights, farm events,
  orders/sales, SAM conversations, BEACON, telemetry, documents, finance status,
  ANALYST, approvals, goals, and trust.
- Every capability declares source of truth, authority tier, freshness, domain,
  argument schema, and failure behavior.
- Tool output uses a common evidence envelope with source, timestamp, confidence,
  freshness, facts, gaps, and safe next reads.

Done when representative questions can gather evidence from at least two domains
without adding a new intent-specific reply function.

### Phase 3 - Adaptive Investigation And Synthesis

- Convert owner language into an executive goal, subjects, questions, constraints,
  and completion criteria.
- Select relevant capabilities dynamically from the registry.
- Inspect results for missing evidence and run bounded follow-up reads.
- Synthesize verified facts, inference, uncertainty, business impact, recommendation,
  actions taken, and anything genuinely required from Charl.
- Never expose `None is None`, raw internal status objects, or unsupported certainty.

Done when real-question scenarios produce useful, evidence-grounded answers even
when wording was not explicitly encoded as a regex intent.

### Phase 4 - Delegation Without Duplication

- Engineering outcomes create or update Supabase CORE missions.
- Farm, sales, and marketing outcomes route to the owning domain capability.
- Delegations include acceptance criteria and are idempotent.
- CHARLIE monitors existing work before creating new work and prevents duplicates.

Done when a broad request can create the correct bounded delegation and report its
verified state without CHARLIE editing code or creating a parallel queue.

### Phase 5 - Durable Executive Memory And Commitments

- Persist goals, active subjects, owner preferences, decisions, commitments,
  delegated work, due checks, outcomes, and learned corrections.
- Operational facts are references to authoritative records, not copied truth.
- Resume naturally across Telegram, dashboard, restart, and later voice sessions.

Done when a follow-up question retains goal/subject and a restart does not lose an
open commitment.

### Phase 6 - Authority And Owner Work Reduction

- Classify reads, routine reversible actions, consequential actions, and red-zone actions.
- Bundle related genuine decisions into one concise owner request.
- CHARLIE handles mechanical CORE blocks and routine review when deterministic
  evidence and delegated policy allow it.
- No customer send, public post, payment, reservation, destructive migration, or
  lifecycle write proceeds without the required authority and fresh-state validation.

Done when false owner escalations are below the acceptance threshold and safety
tests show zero authority bypasses.

### Phase 7 - Outcome Monitoring And Proactive Executive Loop

- Monitor commitments, CORE queue health, domain goals, exceptions, and results.
- Recheck delegated outcomes on schedule.
- Escalate only after bounded recovery is exhausted or real discretion is required.
- Feed outcomes into capability trust and ANALYST validation.

Done when a delegated task is followed through to outcome without a new owner prompt.

### Phase 8 - One Executive Across Interfaces

- Dashboard and private Telegram call the same executive turn API.
- Add voice input/output only as another interface over the same runtime.
- Provide an executive timeline: asked, investigated, decided, delegated, verified.
- Keep detailed evidence behind drill-down, with concise answers by default.

Done when the same question has equivalent evidence and action behavior on every interface.

## Acceptance Scenarios

The release gate must cover at least:

1. "What is happening with CORE?" returns active work, real blockers, recoverable actions, and only genuine owner decisions.
2. "What is happening with pig 104?" reads current pig, weight, litter/location/purpose, and relevant exceptions without guessing.
3. "Can we fulfill Quinton's order profitably?" combines order, availability, pricing, relationship, and delivery evidence.
4. "Why are livestock sales slowing down?" combines SAM, availability, orders, learning, and BEACON evidence and delegates bounded follow-up work.
5. "Make Beacon prepare a livestock awareness post" delegates a draft, never posts publicly, and monitors owner review.
6. "Fix this CORE problem" creates or updates one bounded mission, avoids duplicates, and monitors it.
7. A vague follow-up such as "why?" retains the active subject and deepens the investigation.
8. A failed read triggers safe alternative evidence or a specific gap, never fabricated truth.
9. A red-zone request creates one informative approval bundle and executes nothing.
10. A restart preserves the open executive goal and commitment.

## Success Measures

- Verified answer rate for supported read questions: at least 98%.
- False permission requests for reads: 0.
- False owner escalations: below 2%.
- Delegations duplicated in Supabase: 0.
- Open commitments lost after restart: 0.
- Unauthorized red-zone actions: 0.
- Deterministic required tests: 100% pass.
- Owner corrections accepted into durable preferences and later behavior: at least 95%.
- Executive replies requiring Charl to inspect raw system data: below 10%.
